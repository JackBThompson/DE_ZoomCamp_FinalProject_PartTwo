# CRM Data Migration Pipeline

## Problem Description

Enterprise companies migrating onto a modern SaaS data stack commonly arrive with customer data spread across disconnected legacy systems — a CRM export here, a billing system there, a marketing platform in between. There is no unified view of the customer until someone builds one. This pipeline solves that problem by ingesting five source files that simulate a real-world CRM migration, cleaning and resolving data quality issues with PySpark, modeling a star schema in BigQuery with dbt, and surfacing deal pipeline health and customer growth trends in a Looker Studio dashboard.

The project implements a full ELT pipeline:

- **Extract** — five raw CSV files simulate messy legacy CRM exports (customers, contacts, deals, activities, subscriptions) and are landed in Google Cloud Storage
- **Load** — PySpark cleans, deduplicates, and standardizes the raw CSVs before loading them into BigQuery raw tables
- **Transform** — dbt builds a star schema (fact_deals, dim_customers, dim_subscriptions) from the raw tables and powers the dashboard

---

## Architecture

```
CSV Files (5 sources — customers, contacts, deals, activities, subscriptions)
  → Airflow DAG (orchestration — 3 steps run automatically)
      Step 1: Upload CSVs to GCS  gs://bucket/raw/crm/YYYY-MM-DD/
      Step 2: PySpark (clean, deduplicate, standardize dates, resolve mismatched IDs)
      Step 3: dbt (staging models → core star schema)
  → Cloud - BigQuery Raw Tables (partitioned + clustered)
  → Cloud - BigQuery Analytics Tables (fact_deals, dim_customers, dim_subscriptions)
  → Looker Studio (dashboard)
```

## Why Tables Are Partitioned and Clustered

All raw and core tables are partitioned by `ingestion_date` and clustered by relevant dimension fields (e.g. `plan_tier`, `deal_stage`).

- **Partitioning** — dashboard queries always filter by date range. Partitioning means BigQuery only scans the relevant days rather than the entire table, reducing query cost significantly.
- **Clustering** — most filters are plan or stage specific. Clustering lets BigQuery skip non-matching blocks entirely at no extra cost.

## Steps to Reproduce

All steps run from your terminal. Steps 1–10 run locally or in a Codespace. Steps 11–13 provision and configure the GCP VM. Step 14 triggers the end-to-end pipeline via Airflow. If you run into any problems reproducing, shoot me an email at **etl.jackt@gmail.com** and I'll help troubleshoot.

### 1. Clone the repo
```bash
git clone https://github.com/JackBThompson/DE_ZoomCamp_FinalProject_PartTwo.git
cd DE_ZoomCamp_FinalProject_PartTwo
```

### 2. Install dependencies
```bash
pip install -r requirements.txt
```

### 3. Install Terraform
```bash
wget -q -O /tmp/terraform.zip https://releases.hashicorp.com/terraform/1.7.5/terraform_1.7.5_linux_amd64.zip
unzip -o /tmp/terraform.zip -d /tmp
sudo mv /tmp/terraform /usr/local/bin/
terraform version
```

### 4. Create a GCP project and service account
1. Go to [console.cloud.google.com](https://console.cloud.google.com) and create a new project under 'No Organization'
2. Enable billing on the project — **GCP offers a $300 free credit** for new accounts which covers this project entirely
3. Go to **IAM → Service Accounts → Create Service Account**
4. Add roles: BigQuery Admin, Storage Admin, Compute Admin, Service Account Admin, Project IAM Admin
5. Go to **Keys → Add Key → JSON** and download the key file
6. Place it at the project root and rename it `gcp-key.json`

> **Note:** Do NOT commit `gcp-key.json` to git — it is in `.gitignore`.
>
> **Note:** Do NOT create a GCS bucket or BigQuery dataset manually — Terraform creates them in Step 6.

### 5. Enable required GCP APIs
In the GCP Console under **APIs & Services**, enable:
- Identity and Access Management (IAM) API
- Cloud Resource Manager API
- Compute Engine API

### 6. Authenticate with GCP
```bash
gcloud auth activate-service-account --key-file=gcp-key.json
gcloud config set project YOUR_PROJECT_ID
```

### 7. Provision infrastructure (Terraform)
```bash
cd terraform/
terraform init
terraform apply
cd ..
```

Terraform creates the GCS bucket, BigQuery dataset, Airflow VM, and service account with IAM roles. Note the `airflow_vm_ip` from the output — you'll need it in Step 13.

### 8. Set environment variables
```bash
cp .env.example .env
# Fill in:
# GCS_BUCKET=your-bucket-name       ← from Terraform output
# GCP_PROJECT_ID=your-project-id
# BIGQUERY_DATASET=your-dataset-name
```

Load the variables:
```bash
export $(grep -v '^#' .env | xargs)
```

### 9. Generate CRM data
```bash
python3 scripts/generate_fake_data.py
```
Generates 5 CSV files in `data/raw/` — ~1,700 total records with intentional data quality issues (duplicate emails, mismatched IDs, inconsistent date formats).

### 10. Create BigQuery tables and views
```bash
bq query --use_legacy_sql=false < bigQuerySQL/dashboard_views.sql
```

### 11. Copy key and clone repo onto the VM
```bash
gcloud compute scp gcp-key.json airflow-vm:~/gcp-key.json --zone=us-east4-a

gcloud compute ssh airflow-vm --zone=us-east4-a
```

Once on the VM:
```bash
git clone https://github.com/JackBThompson/DE_ZoomCamp_FinalProject_PartTwo.git
cd DE_ZoomCamp_FinalProject_PartTwo
mv ~/gcp-key.json .
cp .env.example .env
# Fill in .env with your GCP values
```

### 12. Install Spark and Docker Compose on the VM
Still on the VM:
```bash
# Install Spark 3.5.1
wget https://archive.apache.org/dist/spark/spark-3.5.1/spark-3.5.1-bin-hadoop3.tgz
tar -xzf spark-3.5.1-bin-hadoop3.tgz
echo 'export SPARK_HOME=$HOME/spark-3.5.1-bin-hadoop3' >> ~/.bashrc
echo 'export PATH=$SPARK_HOME/bin:$PATH' >> ~/.bashrc
source ~/.bashrc

# Download GCS connector JAR
wget https://storage.googleapis.com/hadoop-lib/gcs/gcs-connector-hadoop3-latest.jar -P ~/

# Install Docker Compose v2
sudo curl -SL https://github.com/docker/compose/releases/download/v2.24.0/docker-compose-linux-x86_64 \
  -o /usr/local/bin/docker-compose
sudo chmod +x /usr/local/bin/docker-compose

# Start Docker
sudo systemctl start docker
```

### 13. Start Airflow on the VM
Still on the VM:
```bash
export GCP_PROJECT_ID=your-project-id
export GCS_BUCKET=your-bucket-name
export BIGQUERY_DATASET=your-dataset-name
export GCP_REGION=us-east4
export AIRFLOW_UID=50000
export POSTGRES_USER=airflow
export POSTGRES_PASSWORD=airflow
export POSTGRES_DB=airflow

sudo -E docker-compose -f docker/docker-compose.yml up -d
```

Airflow initializes in ~60 seconds. Open an SSH tunnel from your local terminal to access the UI:
```bash
gcloud compute ssh airflow-vm --zone=us-east4-a -- -L 8080:localhost:8080
```

Then go to `http://localhost:8080` — login: `admin` / `admin`.

### 14. Trigger the DAG
In the Airflow UI, find the `crm_ingestion` DAG and click **Trigger DAG**.

This runs the full end-to-end pipeline automatically:
1. Uploads all 5 CSVs to GCS
2. Runs PySpark — cleans, deduplicates, loads to BigQuery raw tables
3. Runs dbt — builds staging models and core star schema

### 15. Verify data landed in BigQuery
```bash
bq query --use_legacy_sql=false 'SELECT COUNT(*) FROM `YOUR_PROJECT_ID.YOUR_DATASET.raw_customers`'
bq query --use_legacy_sql=false 'SELECT COUNT(*) FROM `YOUR_PROJECT_ID.YOUR_DATASET.raw_deals`'
```

## Dashboard

To recreate the dashboard, follow `lookerDashboard/looker_setup.md`. To view screenshots, refer to `lookerDashboard/screenshots`.

Built on two tiles that answer the two questions every customer-facing team cares about: **"What does the pipeline look like?"** and **"Is the customer base growing?"**

- **Deal Distribution by Stage** — categorical bar chart showing deal count and total value across pipeline stages (prospecting → closed won/lost). Mirrors what a sales pipeline view looks like in Salesforce or HubSpot — immediately recognizable to any business stakeholder.
- **Customer Signups Over Time** — time series showing new customers per month broken down by plan tier. The tier breakdown surfaces the revenue growth story: when did enterprise customers start coming in vs. free tier?

## Known Limitations

- CSV data is generated with a Python script — in a real migration, source files would arrive via Salesforce exports, HubSpot API, or SFTP drops
- Airflow runs on a single GCE VM — not production-grade but sufficient for this project scope
- Spark runs on the same VM as Airflow — in production these would be separate clusters
